## Context

The `signing` module has placeholder domain types (`SignRequest`, `SignSession`, `SignatureStatus`,
`EsignProvider`) but persists nothing — `SignRequest.agreementId` is a `String` that refers to no
real row. `persistence-foundation` laid the Flyway base (`V1` comment-only baseline,
`ddl-auto: validate` everywhere, `clean-disabled`) but deliberately shipped no feature schema and
no entity. This change adds the first one: a thin, status-less rental `Agreement` so `agreementId`
becomes a real, persisted identity that the signing slices (C: create-signing-request onward) can
build on.

Decisions 1 (status placement) and 2 (content depth) below were brainstormed and agreed with the
user before drafting; the alternatives are recorded for the design trail, not to reopen them.

## Goals / Non-Goals

**Goals:**
- A persisted `Agreement` aggregate with a stable, app-assigned UUID identity and an audit timestamp.
- A minimal but *meaningful* rental-content shape (six typed fields) so create/fetch returns real data.
- The first feature migration (`V2__agreement.sql`) and the first genuine exercise of `ddl-auto: validate`
  against a mapped entity (graduating `persistence-foundation`'s transitional zero-entity posture).
- Create + fetch JSON API, reachable through the real default-deny security filter chain.
- Module boundaries intact (`ModularityTests` green); aggregate internals stay inside `signing`.

**Non-Goals:**
- The signing status FSM (`DRAFT → … → SIGNED`) — owned by the `create-signing-request` change.
- PDF/template rendering, e-stamping, any `documents`-module interaction.
- A link from `Agreement` to `SignRequest`/`SigningRequest` (the signing row does not exist yet).
- List/update/delete endpoints; pagination; soft-delete.
- A lease `startDate`, embedded value objects, or a `jsonb` content blob (all deferred — see D2/D4).

## Decisions

### D1 — Status-less Agreement; the signing FSM lives elsewhere (Model B)
The `Agreement` aggregate carries **no `SignatureStatus`**. An Agreement is a stable contract
document; a signing *attempt* is a separate concern that gets its own aggregate (`SigningRequest`)
and owns the FSM in the `create-signing-request` change.

- *Why:* The existing `SignatureStatus` enum straddles two lifecycles — `DRAFT`/`PDF_GENERATED`
  describe the document, `SIGN_REQUESTED → SIGNED|FAILED|EXPIRED` describe a signing attempt.
  Putting the whole chain on `Agreement` (Model A) conflates the two and breaks the moment a signer
  retries an `EXPIRED` attempt or a second party signs (status would be mutated/overwritten,
  per-attempt history lost). Keeping `Agreement` status-less avoids a dead, always-`DRAFT` column
  and does not pre-commit the FSM's home.
- *Alternatives:* **Model A** (one FSM on `Agreement`) — simplest, one row, but conflates document
  with attempt; rejected. **Model C** (split: `Agreement` owns `DRAFT→READY`, `SigningRequest` owns
  the rest) — most correct but over-engineered for a thin prereq; not now.
- *Hands to slice C:* whether `DRAFT`/`PDF_GENERATED` should be refactored off `SignatureStatus`
  onto the document side (a `SigningRequest` arguably doesn't exist until signing is requested).
  Recorded as an open question, not resolved here.

### D2 — Minimal typed content: six flat columns
`Agreement` holds `landlordName`, `tenantName`, `propertyAddress`, `monthlyRent`,
`securityDeposit`, `termMonths` as flat columns on the entity (no embedded value objects yet).

- *Why:* A rental agreement's *core identity* fields are stable across Indian states; the
  jurisdiction-variable parts are clauses/stamp-duty/rendering, which belong to `documents`/`rules`
  later — not to this aggregate. Typed columns give a real validation surface and real columns for
  the `validate` graduation (D5). Explicit over implicit.
- *Alternatives:* **Bare metadata** (id + title) — `agreementId` would barely mean anything and the
  migration/validate path would have nothing real to exercise; rejected. **`jsonb` blob** — flexible
  but untyped, hides the model, and defeats the typed-`validate` exercise; deferred until the rules
  engine actually needs variable schema. **Full DDD** (embedded `Landlord`/`Tenant`/`Money`/`Term`
  value objects) — cleaner domain but disproportionate code for a thin prereq; add value objects
  when behavior attaches.
- *Deferred field:* a lease `startDate` is arguably more fundamental than `securityDeposit`, but it
  does not matter until dates feed PDF/stamp-duty in slice E, so `termMonths` alone represents the
  term for now.

### D3 — Application-assigned UUID identity
The id is a `UUID` generated by the aggregate's factory (e.g. `UUID.randomUUID()`), not a database
`gen_random_uuid()` default or a sequence. The column is `uuid PRIMARY KEY`.

- *Why:* Identity exists the instant the aggregate is constructed — testable in pure unit tests with
  no DB round-trip, and no reliance on a DB default. Avoids the read-after-write needed to learn a
  DB-generated id. UUID (not a serial) keeps ids non-enumerable and is friendly to a future
  distributed/sharded world.
- *Alternatives:* DB `gen_random_uuid()` default (core in PG13+) — works, but forces a DB round-trip
  to learn the id and couples identity to the DB; rejected. Auto-increment `bigserial` — enumerable,
  leaks volume; rejected for an identity that appears in URLs.

### D4 — Money as `BigDecimal` / `numeric(12,2)`
`monthlyRent` and `securityDeposit` map to `numeric(12,2)` columns and `BigDecimal` in Java; both
are bean-validated `@PositiveOrZero` (`>= 0`). `termMonths` is an `int` column, `@Positive` (`> 0`).

- *Why:* `numeric` is exact (no binary-float rounding on money) and `(12,2)` comfortably covers
  rents/deposits in INR. `BigDecimal` keeps arithmetic exact end-to-end.
- *Alternatives:* paise-as-`BIGINT` — exact but ugly and error-prone at the API boundary; rejected.
  `double` — never for money.

### D5 — First entity-bearing migration; `validate` is now genuinely exercised
`V2__agreement.sql` (forward-only) creates the `agreement` table to match the entity mapping exactly
— column names, `numeric(12,2)`, `not null`, `uuid` PK, `timestamptz created_at`. The integration
test asserts both that a row persists/fetches **and** that the context boots under `ddl-auto:
validate` against `V2`. This supersedes `persistence-foundation`'s transitional
"validate-green-with-zero-entities" requirement (removed in this change's `backend-persistence`
delta).

- *Why:* `validate` and the Flyway-before-JPA ordering were only *tolerated* against an empty schema
  before; with a mapped entity they are first truly tested. A mismatch (e.g. entity column absent in
  the migration) now fails context startup — exactly the guardrail we want.
- *Mechanics:* no schema/catalog override; the table lands in the default schema Flyway migrates and
  Hibernate validates. Naming follows the convention from `persistence-foundation`
  (`V<n>__<snake_case>.sql`, `classpath:db/migration`).

### D6 — Endpoints permitted in the default-deny baseline, verified through the real filter chain
`SecurityConfig` (the `backend-security-baseline` default-deny chain) gains `permitAll` for
`/api/agreements/**`. The integration test runs against the **full application context** so requests
traverse the real Spring Security filter chain.

- *Why:* The default-deny baseline returns `403` for any unlisted path; without a permit rule the
  endpoints are unreachable. A `@WebMvcTest`/standalone MockMvc slice does **not** load the security
  filter chain, so it would pass even if the permit rule were missing — the exact false positive hit
  during `spring-security-baseline`. The authoritative test must exercise the real chain.
- *Note:* `permitAll` is correct for this sandbox (no auth yet); real authz arrives with identity/KYC.

### D7 — Aggregate internals stay inside `signing`
The `Agreement` entity, its repository, and the create/fetch service logic are package-private
within `in.agreementmitra.signing`; only the `api` controller and its DTO records are reachable as
the module's surface. Nothing about `Agreement` is exported to other modules yet.

- *Why:* Preserves Spring Modulith boundaries (`ModularityTests` stays green) and matches the
  convention "`public` only on module API." Cross-module exposure (events, a published lookup port)
  is added when another module actually needs it — not speculatively.

### D8 — Date/time handling: `createdAt` is an instant (UTC); lease dates deferred
This change captures exactly one temporal field, `createdAt`, an **instant** (a moment in time) —
stored UTC as `timestamptz` / Java `Instant`, serialized ISO-8601 with `Z`
(`2026-06-20T12:31:23Z`); the future SPA renders it in the viewer's timezone. No lease **calendar
dates** (`startDate`/`endDate`) are captured here (see D2 deferral).

- *Why the distinction matters:* an instant is timezone-relative; a calendar date is not. A lease
  "starts 1 July 2026" is the same day for every viewer — storing it as a UTC timestamp and
  tz-converting for display would shift the day for viewers off UTC (an off-by-one bug). When lease
  dates land (slice E) they SHALL be `date` / `LocalDate`, stored and shown as-is, never tz-converted.
- *App-wide policy* (codified in CLAUDE.md Conventions): instants → UTC + viewer-tz render; calendar
  dates → `LocalDate`, no conversion; one shared UI date format unless explicitly customized. The UI
  half is a frontend concern (no SPA in this backend-only change).
- *This change:* `createdAt` already follows the policy; nothing extra to build. The D5 round-trip
  test confirms it persists and returns unchanged.

## Risks / Trade-offs

- **Entity/migration drift** (entity and `V2__agreement.sql` hand-written separately could diverge)
  → the `validate`-against-`V2` integration test (D5) fails context startup on any mismatch, so
  drift is caught at test time, not in prod.
- **Status-less now, FSM later** (D1 leaves the document/attempt split for slice C) → accepted and
  explicit; the open question is recorded for `create-signing-request`. No code here pre-commits it.
- **`permitAll` on a write endpoint** (D6 — unauthenticated create in the sandbox) → bounded by the
  repo's sandbox + dummy-data-only constraint; flagged to revisit when authz lands. No PII or secret
  is involved (see proposal's PII/security review).
- **Deferred `startDate`** (D2) → a later CR adds the column via `V3`; forward-only migrations make
  that a clean additive change, so deferring costs nothing now.

## Migration Plan

1. Add `V2__agreement.sql` creating the `agreement` table.
2. Add the `Agreement` entity + repository + DTO records + service + `api` controller in `signing`.
3. Add the `permitAll` rule for `/api/agreements/**` to `SecurityConfig`.
4. Verify: `./run-tests.sh check` — Flyway applies `V1`+`V2`, context boots under `validate`,
   persist/fetch + endpoint integration tests pass, `ModularityTests` green, `securityScan` clean.
   No `gradle.lockfile` change expected (no new dependencies).

Rollback: forward-only — a corrective `V3` would be shipped rather than editing `V2`. This change
adds only additive schema and new code, so rollback before release is simply reverting the commit
(the `agreement` table is new and unreferenced elsewhere).

## Open Questions

- (For `create-signing-request`, not this change) Should `DRAFT`/`PDF_GENERATED` be refactored off
  `SignatureStatus` onto the document/agreement side, leaving `SignatureStatus` as the signing-attempt
  FSM (`SIGN_REQUESTED → SIGNED|FAILED|EXPIRED`)?
- (Future) Does `Agreement` need its own editing lifecycle (`DRAFT → FINALIZED`) to gate "only
  request signing on a finalized agreement"? Deferred until slice C demonstrates the need.
