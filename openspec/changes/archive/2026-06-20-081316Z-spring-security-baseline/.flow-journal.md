## manual-test — 2026-06-20  [⛔→resolved]
Outcome: User manually ran the live app (bootRun) and found BOTH permitted POSTs returning 403 instead of reaching the controller — a REAL defect MockMvc had masked.
Halts: Manual-test gate caught a defect. Treated as a finding under Issue policy → folded a minimal in-scope fix + corrected test fidelity; did NOT archive over the break.
ROOT CAUSE (found via a real-port TestRestTemplate diagnostic + Spring Security TRACE): the webhook/signing endpoints WERE permitted and reached the throwing stub, but Spring Security 6 re-authorizes Boot's internal dispatch to /error, which hit anyRequest().denyAll() → 403, masking the real 500. NOT CSRF (CsrfFilter confirmed absent), NOT the matcher.
FIX: requestMatchers("/error").permitAll() in SecurityConfig + server.error.include-* pinned off (so reachable /error leaks nothing). Reverted a red-herring method-agnostic matcher change back to HttpMethod.POST.
TEST FIDELITY: replaced the MockMvc test (false-positive: MockMvc rethrows handler exceptions, skips the /error dispatch) with a real-port @SpringBootTest(RANDOM_PORT)+TestRestTemplate test against the Testcontainers harness (Docker-gated, skips w/o Docker). Deleted the throwaway diagnostic. Real-container result now: webhook/signing POST → 500 (reach stub), health 200, unmapped/list/env 403. Full ./gradlew check green; security test 7/7, 0 skipped.
Decisions: chose permitAll("/error") (visible, standard) over excluding ERROR from the security filter dispatcher-types. Accepted Docker-gated test (fidelity > no-Docker convenience).
Modified files: SecurityConfig.java, application.yml, SecurityBaselineIntegrationTest.java (rewritten), deleted SecurityBaselineRealPortDiagnosticTest.java; artifacts proposal/design/specs/tasks updated for the /error permit + test approach.
