# agreement Specification

## Purpose
The rental-agreement aggregate: a stable, persisted contract document with an
application-assigned identity and typed rental content, plus its create + fetch API. Status-less by
design — the signing lifecycle (request → sign → complete) is a separate capability. This is the
first entity-bearing capability and is where Flyway-managed schema and `ddl-auto: validate` are
genuinely exercised against a real mapped entity.

## Requirements
### Requirement: Agreement aggregate and persisted shape
The `signing` module SHALL define an `Agreement` aggregate, persisted as a single row in an
`agreement` table. Each Agreement SHALL have an application-assigned UUID identity (generated by
the aggregate itself, not by a database default or sequence) and a `createdAt` timestamp set once
at creation. An Agreement SHALL hold exactly six rental-content fields: `landlordName`,
`tenantName`, and `propertyAddress` (non-blank text); `monthlyRent` and `securityDeposit`
(non-negative decimal money amounts); and `termMonths` (a positive integer). The aggregate SHALL
carry **no signing status** — the signing lifecycle FSM is owned by a separate capability and is
not part of the Agreement.

#### Scenario: Agreement is created with a stable identity
- **WHEN** an Agreement is constructed from valid rental content
- **THEN** it has a non-null UUID id assigned by the aggregate
- **AND** it has a `createdAt` timestamp
- **AND** the same id is returned by every later fetch of that Agreement

#### Scenario: Aggregate carries no signing status
- **WHEN** the `Agreement` aggregate is inspected
- **THEN** it exposes no `SignatureStatus` or signing-lifecycle field
- **AND** persisting and fetching an Agreement involves no signing state

### Requirement: Create an agreement
The system SHALL expose `POST /api/agreements` accepting a JSON body of the six rental-content
fields. On valid input it SHALL persist a new Agreement and respond `201 Created` with a JSON body
containing the generated id, the stored fields, and `createdAt`. The request SHALL be
bean-validated; on any validation failure it SHALL respond `400 Bad Request` and persist no row.

#### Scenario: Valid request creates an agreement
- **WHEN** a client POSTs valid landlord/tenant/property/rent/deposit/term content
- **THEN** the response is `201 Created` with the new id and the stored fields
- **AND** a corresponding `agreement` row exists

#### Scenario: Blank required text is rejected
- **WHEN** a client POSTs with a blank `landlordName`, `tenantName`, or `propertyAddress`
- **THEN** the response is `400 Bad Request`
- **AND** no `agreement` row is created

#### Scenario: Out-of-range numbers are rejected
- **WHEN** a client POSTs with a negative `monthlyRent` or `securityDeposit`, or a `termMonths` of zero or less
- **THEN** the response is `400 Bad Request`
- **AND** no `agreement` row is created

### Requirement: Fetch an agreement
The system SHALL expose `GET /api/agreements/{id}` returning the Agreement with that id as JSON
(`200 OK`). If no Agreement exists for the id, it SHALL respond `404 Not Found`.

#### Scenario: Fetch an existing agreement
- **WHEN** a client GETs `/api/agreements/{id}` for a persisted Agreement
- **THEN** the response is `200 OK` with the id, all six fields, and `createdAt`

#### Scenario: Fetch a missing agreement
- **WHEN** a client GETs `/api/agreements/{id}` for an id that does not exist
- **THEN** the response is `404 Not Found`

### Requirement: Agreement endpoints are reachable under default-deny security
The default-deny Spring Security baseline SHALL permit the agreement create and fetch endpoints so
they are reachable without authentication in this sandbox. This SHALL be verified through the real
security filter chain (a full-context integration test), not a filter-blind controller slice.

#### Scenario: Create and fetch pass the security filter chain
- **WHEN** the agreement endpoints are exercised through the application's real security filter chain
- **THEN** requests are not rejected with `401`/`403` by the default-deny baseline
- **AND** they reach the controller and return their documented status codes

### Requirement: Schema matches the agreement entity mapping
The `V2__agreement.sql` migration SHALL create the `agreement` table such that JPA
`ddl-auto: validate` passes against the `Agreement` entity mapping: every mapped column exists with
a compatible type and nullability. This is the first entity-bearing migration and is where the
Flyway-before-JPA ordering and `validate` are genuinely exercised (superseding the transitional
zero-entity posture in `backend-persistence`).

#### Scenario: Validate passes against the migration
- **WHEN** the application context boots with `V2` applied and `ddl-auto: validate`
- **THEN** the context starts with no schema-validation failure for the `Agreement` entity

#### Scenario: A persisted agreement round-trips through the migration schema
- **WHEN** an Agreement is saved and then loaded from the `V2`-created `agreement` table
- **THEN** all six fields, the id, and `createdAt` are returned unchanged
